# Agent Guidance

This file provides context for AI agents (Claude Code, Copilot, etc.) working in this repository.

## What this repo is

GitOps configuration for a homelab Kubernetes cluster managed by [Flux CD](https://fluxcd.io).
The cluster runs on Talos Linux. Infrastructure is declared here; Flux reconciles the cluster state to match.

## Repository layout

```
clusters/production/     # Flux entrypoint — bootstrap and Kustomization objects
  flux-system/           # Auto-generated by `flux bootstrap` — do not edit manually
  infrastructure.yaml    # Flux Kustomization: deploys infrastructure/
  configs.yaml           # Flux Kustomization: deploys infrastructure/configs/
  apps.yaml              # Flux Kustomization: deploys apps/

infrastructure/          # Cluster infrastructure (operators, CRDs, controllers)
  networking/            # Network controllers: metallb, gateway-api, nginx-gateway
  storage/               # Storage controllers: democratic-csi
  security/              # Security/PKI controllers: cert-manager
  configs/               # Custom resources that configure the above operators
    metallb/             # IPAddressPool, L2Advertisement
    gateway/             # Gateway, HTTPRoutes, namespace
    cert-manager/        # ClusterIssuer, Certificate, SOPS-encrypted Cloudflare token

apps/                    # Application workloads
  prod/                  # Production namespace (prod)
  dev/                   # Development namespace (dev)

talos/                   # Talos Linux OS configuration (separate from Flux)
```

## Deployment ordering (Flux dependency chain)

```
flux-system → infrastructure → infrastructure-configs → apps
```

- **infrastructure**: installs operators and CRDs (HelmReleases). Must come first so CRDs exist.
- **infrastructure-configs**: configures those operators (IPAddressPool, ClusterIssuer, Gateway, etc.). Depends on infrastructure.
- **apps**: application workloads. Depends on infrastructure-configs (gateway and certs must be ready).

Do not break this ordering. If you add a new Flux Kustomization object, set `dependsOn` appropriately.

## Adding infrastructure (operators/controllers)

1. Create a directory under the appropriate category:
   - `infrastructure/networking/` for network operators
   - `infrastructure/storage/` for storage operators
   - `infrastructure/security/` for security/PKI operators
2. Add `namespace.yaml`, `repo.yaml` (HelmRepository), `release.yaml` (HelmRelease), and `kustomization.yaml`
3. Reference the new directory in `infrastructure/kustomization.yaml`
4. If the operator needs configuration (CRD instances), add a matching directory under `infrastructure/configs/` and reference it in `infrastructure/configs/kustomization.yaml`

## Adding applications

See README.md for the full walkthrough. Quick summary:
- Raw manifests: create `apps/prod/<appname>/` with deployment, service, httproute, kustomization.yaml; register in `apps/prod/kustomization.yaml`
- Helm-based: create repo.yaml + release.yaml; add httproute if needed
- Label the app namespace with `gateway-access: shared` to allow routing through the shared gateway

## Secrets and SOPS encryption

All secrets are encrypted with [SOPS](https://github.com/getsops/sops) using an age key.

**Rules (defined in `.sops.yaml`):**
- Only files matching `*.sops.yaml` are encrypted
- Only `data` and `stringData` fields are encrypted (Kubernetes Secret fields)

**To create a new secret:**
```bash
# Create the plain yaml first, then encrypt in place
sops --encrypt --in-place path/to/secret.sops.yaml
```

**To edit an existing secret:**
```bash
sops path/to/secret.sops.yaml
```

**Never commit unencrypted secrets.** Files named `*.sops.yaml` must always be SOPS-encrypted before committing.

## Gateway and TLS

- **Gateway class**: `nginx` (NGINX Gateway Fabric)
- **Shared gateway**: `shared-gateway` in namespace `ingress-gateway`
- **TLS**: wildcard cert `wildcard-va1-uk-tls` covering `*.va1.uk`, issued by Let's Encrypt via Cloudflare DNS-01
- **To expose an app**: create an HTTPRoute in the app's namespace with `parentRefs` pointing to `shared-gateway` in `ingress-gateway`
- **Namespace must** have label `gateway-access: shared` for the gateway to accept routes from it

## Storage

- **Storage class**: `iscsi` (democratic-csi, backed by TrueNAS at `tower.bigpri.me`)
- **Reclaim policy**: Delete
- PVCs referencing `iscsi` are provisioned via iSCSI from the TrueNAS ZFS pool `storage/k8s-ds/v`

## What NOT to do

- Do not edit `clusters/production/flux-system/gotk-components.yaml` — it is auto-generated by `flux bootstrap`
- Do not add resources directly to `clusters/production/flux-system/` unless bootstrapping
- Do not commit plain-text secrets — always encrypt with SOPS first
- Do not place application configs (CRD instances) in `infrastructure/networking|storage|security/` — those are for HelmReleases only; configs go in `infrastructure/configs/`
- Do not add apps to `apps/dev/` without first creating a namespace with the `gateway-access: shared` label if HTTP routing is needed

## Talos

Talos configuration lives in `talos/`. It is managed separately from Flux.
Secrets in `talos/` are encrypted with SOPS using a more aggressive regex (see `talos/.sops.yaml`).
See `talos/README.md` for apply/upgrade procedures.

## Flux CLI quick reference

```bash
# Check reconciliation status
flux get all -A

# Force reconcile
flux reconcile kustomization infrastructure --with-source
flux reconcile kustomization infrastructure-configs --with-source
flux reconcile kustomization apps --with-source

# Check a specific HelmRelease
flux get helmrelease -A

# Check events for failures
flux events -A
```
